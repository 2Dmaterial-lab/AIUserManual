# 前言：AI辅助科研的理念与心态

## AI不是替代，是放大器

AI 工具的本质是**放大你的能力**，而不是替代你的思考。就像计算器没有替代数学家，而是让数学家从繁琐的手算中解放出来，AI 让你从重复性劳动中解放出来，把精力留给真正需要创造力的事情：

- 数据处理、格式转换、批量出图 → 让 AI 做
- 实验设计、物理直觉、创新思路 → 你来做
- 代码框架生成、语法查错、bug 定位 → 让 AI 做
- 算法选择、结果判断、物理意义解读 → 你来做

### 2026年AI新范式：从「对话」到「智能体」

AI 工具在过去一年发生了质变，值得了解三个新概念：

1. **推理模型（Reasoning Models）**：GPT-5.5、Claude Opus 4.7、Gemini 3.1 Pro、DeepSeek V4、GLM-5.1 等模型都更强调复杂任务、工具使用和多步推理。面对公式推导、疑难 debug、多约束方案比较时，**开启推理/思考模式**通常比普通快速回答更稳。

2. **AI Agent（智能体）**：从「你问它答」升级为「给定目标，AI 自主规划并执行」——能读写文件、运行命令、搜索网络。代表产品如 Claude Code、Codex CLI、Cursor Agent、Windsurf Cascade。对于多文件代码重构、批量数据处理，Agent 可以自主完成闭环。

3. **MCP（模型上下文协议）**：由 Anthropic 发布并被多类 AI 工具支持的开放协议。它让 AI 以标准方式连接本地文件、数据库、API 等外部工具。对课题组而言，Claude Desktop / Cursor / Claude Code + MCP 可以让 AI **在授权范围内读取本地论文、代码或数据文件**，减少复制粘贴。

> 这些新范式让 AI 从「聊天工具」进化为「工作伙伴」。但对于科研，核心原则不变：**你始终是结果的最终责任人**。

## 提问的艺术：如何高效地向AI提问

AI 的输出质量，很大程度上取决于你的输入质量。掌握以下原则，能让 AI 的回答从"泛泛而谈"变成"精准可用"。

### 原则1：给上下文

**差的提问**：
> 帮我画个图

**好的提问**：
> 我用 MATLAB 分析了一组光学透过率数据，波长范围 400-800nm，想画一张 publication 级别的图，要求：x轴是波长(nm)，y轴是透过率(%)，线宽1.5pt，字体Helvetica 12pt，不要网格线，输出为300dpi的EPS格式。

差异在于：好的提问交代了**工具**、**数据类型**、**具体要求**、**输出格式**。

### 原则2：给示例

如果你想要特定风格的输出，给 AI 一个示例比千言万语更有效：

> 请按照以下格式输出结果：
> ```
> 波长: 450nm, 透过率: 92.3%, 拟合误差: 0.02
> ```

### 原则3：分步走

复杂任务拆成多步，每步确认后再进行下一步：

1. "先帮我写一个读取 CSV 数据的 MATLAB 函数框架"
2. （确认框架后）"现在加上数据的归一化处理"
3. （确认后）"加上绘图功能，要求..."

### 原则4：让AI解释，不只是生成

当你不理解 AI 生成的代码时：

> 请逐行解释这段代码的逻辑，特别是第15-20行，我不理解为什么这里要转置矩阵。

理解代码比得到代码更重要——你需要为结果负责。

### 原则5：迭代优化

第一版输出通常不是最好的。学会迭代：

> 这段代码可以运行，但运行太慢了，数据量大约100万行，请帮我优化性能。

### 原则6：复杂问题开启推理模式

当你面对**多步逻辑推导、复杂公式变换、棘手 bug 排查**时，使用支持推理的模型并明确要求深度思考：

> 下面的问题涉及多层嵌套的偏微分方程和边界条件，请用扩展思考模式，逐步推导，不要跳步。

在 Claude 中开启「扩展思考」、ChatGPT 中使用推理模型、Gemini 中使用面向复杂任务的 Pro / Deep Think 类能力、DeepSeek 中使用思考模式，都能触发这种慢而深入的分析——对于科研场景中的硬问题尤为有效。

## 何时该用AI，何时该自己学

### 应该用AI的场景

- **语法记不清**：API 参数、函数用法，问 AI 比查文档快
- **批量处理**：重复性操作（批量重命名、格式转换、循环出图），可交给 Agent 自主完成
- **报错解读**：把错误信息粘贴给 AI，通常能快速定位原因
- **初稿生成**：代码框架、文档草稿、图表配置
- **跨语言转换**：MATLAB 代码转 Python，VBA 转 Python 等
- **学习新工具**：让 AI 给你一个快速入门教程
- **复杂推理**：公式推导、疑难 bug 分析，开启推理模式让 AI 深度思考
- **多文件改动**：使用 Claude Code 或 Cursor Agent 完成跨文件重构

### 应该自己学的场景

- **核心专业知识**：物理原理、数学推导——AI 可能给错误答案
- **结果验证**：AI 生成的代码必须自己验证，不能盲信
- **领域特定判断**：仿真参数是否合理、实验方案是否可行
- **写作中的原创观点**：论文的核心创新必须是你自己的思考

## 科研诚信：AI使用的边界

### 可以做的

- 用 AI 辅助编写、调试代码
- 用 AI 帮助理解报错信息、优化算法
- 用 AI 辅助翻译、润色论文语言
- 用 AI 生成数据处理脚本、绘图脚本

### 不可以做的

- 让 AI 代写论文核心内容并声称是自己写的
- 直接使用 AI 生成的数据或结果而不加验证
- 将未发表的实验数据、敏感信息输入 AI（尤其是云端服务）
- 用 AI 生成虚假数据或篡改实验结果

### 隐私红线

**绝对不要**将以下内容输入云端 AI 服务：

- 未发表的实验原始数据
- 涉及专利的技术细节
- 包含个人信息的文件
- 课题组未公开的研究成果

如果需要处理敏感数据，考虑以下方案：
- **本地部署开源模型**：使用 Ollama + DeepSeek V4、GLM-5.1、Qwen3 等开源/开放权重模型，数据完全不出本机
- **MCP 本地连接**：Claude Desktop、Cursor 或 Claude Code + 文件系统 MCP server 可在授权目录内读取本地文件，减少复制粘贴；但若使用云端模型，对话与必要上下文仍可能经服务商处理
- **数据脱敏**：对数据进行脱敏处理后再输入（替换真实数值、项目名称等）

## 科研场景中的典型误区（翻车提醒）

下列情况在组内讨论中较常见，**不是**否定 AI，而是提醒：输出必须经过你自己的学科判断与验证。

- **物理常数与单位**：AI 可能给出过时或近似常数，或与文献不一致的单位换算；仿真与公式推导请以教材、官方文档或本领域共识为准，关键数字自行核对。
- **边界条件与网格**：在 COMSOL、FDTD 等工具中，AI 只能给思路；边界是否物理、网格是否足够，必须由你根据模型检查。
- **「能跑」不等于「对」**：代码不报错只说明语法可执行，结果是否反映真实实验或正确模型，需要对照实验、极限情况或文献结果。
- **过度简洁的提示**：只问「帮我画张图」容易得到泛泛结果；参见上文「提问的艺术」，把工具、数据尺度、期刊要求写清楚。

**建议做法**：把 AI 当作需要验收的合作者——关键步骤保留中间结果（数据快照、参数表、一两句推导记录），便于自查与组内交流。
